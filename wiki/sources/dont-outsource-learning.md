# Don't Outsource the Learning

## Summary

This source argues that AI coding tools can either deepen or weaken engineering skill depending on how they are used. Delegating code generation without forming a hypothesis, reading the output, or reconstructing the reasoning creates cognitive debt: tasks close faster, but the engineer's mental model does not improve.

The updated source grounds the warning in recent studies: AI-assisted engineers can complete work quickly while retaining less, LLM-supported writing can reduce later recall of produced work, and early LLM access can anchor the user's problem framing before they have formed their own diagnosis. The shared conclusion is that the tool is not the main variable; the user's posture and order of operations are.

## Key Ideas

- The default AI coding loop is optimized for task closure, not skill formation.
- AI assistance is not inherently harmful; the posture matters. Asking conceptual questions, testing a hypothesis, and reviewing output can support learning, while copy-paste acceptance weakens ownership.
- Early LLM access can anchor the problem framing, so engineers should form their own diagnosis before asking the model.
- Learning-oriented modes such as Socratic questioning, study mode, and guided learning are useful for production engineers learning unfamiliar tools, not just for students.
- Pure delegation breaks down when systems fail, generated answers are confidently wrong, frameworks change, or the problem leaves the median path represented in common training examples.
- Learning-oriented prompts should ask for explanation, alternatives, and tradeoffs before code.
- Engineers should treat AI output like a pull request from a junior engineer: read it, critique it, and be able to explain why it should merge.
- A healthy AI workflow tracks two metrics: what shipped and what the engineer learned.
